Foundations: One Request, One Program, and the Vocabulary

Before we scale anything, you need a concrete, non-magical picture of a single request hitting a single program — and crisp definitions for the words everyone throws around (server, instance, container, HTTP). This section builds that floor. Everything later stands on it.

🎯How to read this guide

Every topic answers four questions — what it is, why it exists, how it works, and when it matters (vs. when it doesn't) — and grounds them in a real scenario. Watch for these coloured boxes:

1.1What "system design" is — and the interview frame everyone uses

What it is. System design is the craft of deciding how the parts of a software system — programs, databases, networks, queues — fit together so the whole thing is fast enough, reliable enough, and cheap enough for the job. Writing a function is programming; deciding that the function should run on twenty machines behind a load balancer, read from a cache, and write to a replicated database is system design.

Why it exists. A single program on a single computer can only do so much. The moment you have more users, more data, or stricter uptime promises than one machine can satisfy, you must spread the work across many moving parts — and those parts can fail, fall behind, or disagree. System design is the body of knowledge for making that sprawl behave.

🌍Real-world scenario

You're asked to "design Twitter." A beginner starts drawing boxes immediately. A system-design thinker first asks: How many users? Read-heavy or write-heavy? Do tweets need to appear instantly or is a few seconds' delay fine? Is losing a draft acceptable? Those answers change every box you'd draw. Stage 1 (clarify) is doing 80% of the work.

1.2What a server actually is (it's not a magic box)

What it is. A server is, at its core, a program that sits and waits for requests, then answers them. That's it. The word is overloaded — people also call the computer running that program a "server" — but the load-bearing meaning is the role: something that listens and responds.

📖Jargon decoder — "server" has two meanings
Server (the role)
A running program that listens for incoming requests and sends back responses — e.g. the Nginx process answering web traffic.
Server (the machine)
A computer dedicated to running such programs — "a server in a rack." When someone says "the server crashed," they might mean the program died or the whole box went down. The ambiguity is real; ask which they mean.

Why it exists. The whole internet runs on a simple division of labour: some programs ask (clients) and some programs answer (servers). Centralizing the "answer" side lets many users share one authoritative copy of data and logic instead of everyone keeping their own.

How it works. A server program binds to a network address and a port (more on ports in 1.6), then loops forever: accept a request → do some work → send a response → wait for the next one. A web server like MDN describes it can be "hardware, software, or both" — but the minimum software piece is an HTTP server that understands the web's request/response language.[MDN]

🔑Key idea

A server is a program waiting for requests. Hold that picture — most of system design is "how do we have enough of these waiting programs, fed with the right data, fast enough, without losing anything?"

1.3The layers inside "a server": web server vs. application server

What it is. When your web app runs, "the server" is usually two cooperating programs: a web server out front and an application server behind it.

📖Jargon decoder
Web server (e.g. Nginx, Apache HTTP Server)
Speaks HTTP fluently. Serves static files (HTML, CSS, images) as-is, handles encryption, and forwards anything dynamic to the app behind it.[MDN]
Application server (your code)
Runs your program to generate a response — querying a database, applying business logic, building the page. MDN calls a dynamic site "a static web server plus extra software, most commonly an application server and a database."[MDN] In practice this is a Python process (run via Gunicorn/Uvicorn), a Node.js process, a Java servlet container, etc.
Reverse proxy
A mode of the web server: it takes the client's request and forwards it to a hidden backend, then relays the backend's answer back. To the client, the reverse proxy is the website. Apache documents this as the gateway role of the same web-server software.[Apache]

Why it exists. Serving a file and running application logic are different jobs. Web servers are extremely good and efficient at the first; your app framework is built for the second. Splitting them lets the web server absorb the raw network load, terminate encryption, cache, and shield your app — which never has to face the open internet directly.[F5/NGINX]

How it works. The web server matches the incoming URL path. Static file? It returns the bytes itself. Dynamic path? It proxies the request to the application server, waits for the generated response, and passes it back to the client.

⚠️Watch out — the terms genuinely overlap

The same software (Nginx, Apache) is simultaneously a "web server" and a "reverse proxy." And some tools (e.g. Tomcat) get called both a "web server" and an "application server." Don't expect crisp, mutually exclusive boxes — these are roles one program can play at once.[Apache mod_proxy]

1.4Client–server and the request/response round-trip

What it is. Almost everything on the web is built on one pattern: a client asks, a server answers — one request, one response. HTTP, MDN notes, is "a client–server protocol: requests are initiated by the client, usually the browser."[MDN]

📖Jargon decoder
Client
The program that initiates the conversation and asks for something — your browser, a mobile app, or even another server calling an API.
Request / Response
The two messages of the exchange. The client sends a request ("give me /home"); the server sends back a response (a status code + the content).
Round-trip
One full there-and-back cycle: request out, response in. Its duration is a big chunk of "how slow does this feel?"

Why it exists. It's a clean contract. The client doesn't need to know how the server produces the answer; the server doesn't need to know who the client is in advance. This decoupling is what lets a browser talk to any website and a server handle any client.

How it works. The client opens a connection, sends a request message, and waits. The server does its work and replies. Then — crucially — the exchange is over; the server doesn't inherently remember you (we'll unpack that in 1.7).

🌍Real-world scenario

You open maps.example.com. Your browser (client) sends a request for the page; the server responds with HTML. That HTML says "I also need 30 map tiles, a stylesheet, and some JavaScript" — so the browser fires 30+ more requests, each its own round-trip. A single "page load" is really a flurry of client–server exchanges. This is why reducing the number and cost of round-trips (the theme of caching and CDNs later) matters so much.

1.5Machine vs. VM vs. instance vs. container vs. process

These five words get used interchangeably and it quietly confuses everyone. Here's the crisp version, from the outside in.

📖Jargon decoder — from hardware to your running code
Machine
A computer — real hardware (CPU, RAM, disk, network) or a virtual equivalent. The most generic term. ("Host" and "box" are casual synonyms.)
Virtual machine (VM)
Software that virtualizes hardware: a hypervisor carves one physical machine into several VMs, each running a full guest operating system with its own kernel. Strong isolation; heavier (images in the tens of GBs; boots like a real computer).[Microsoft]
Instance
In cloud terms, a running VM you provisioned. AWS literally defines an EC2 instance as "a virtual server in the AWS Cloud," launched from a template (an AMI).[AWS] "Instance" stresses one particular running copy.
Container (e.g. Docker)
A lighter box that virtualizes the operating system instead of the hardware: containers "share the machine's OS kernel and therefore do not require an OS per application."[Docker] Result: images in the tens of MBs and startup in seconds — but a weaker security boundary than a VM.
Process
A running program with its own slice of memory, managed by the OS. A container is essentially one (or a few) isolated processes on the shared kernel. Your code, while running, is a process.

Why the distinction matters. "Spin up another instance" means provision another VM (heavier, stronger isolation). "Run another container" means start another lightweight box on an existing machine (cheaper, faster, denser). Knowing which you mean drives cost, speed, and isolation trade-offs you'll make constantly.

How they nest. These aren't competing choices — they stack. Microsoft notes that "many deployments of containers use VMs as the host… especially in the cloud."[Microsoft]

✓ Reach for a VM / instance when

  • You need a strong security boundary or a different OS than the host.
  • You're running long-lived, heavyweight workloads.
  • You want the simplest mental model: "a computer in the cloud."

✓ Reach for a container when

  • You want fast startup and to pack many copies densely on one machine.
  • You need "works on my machine" to also mean "works in production" (the image carries the dependencies).
  • You're scaling stateless app copies up and down constantly.
⚠️Watch out

Docker markets "the strongest default isolation in the industry" — that's a comparison among container tools, not a claim that containers isolate better than VMs. Microsoft is explicit: a VM gives a stronger security boundary because each VM has its own kernel.[Microsoft]

1.6Anatomy of a web request, end to end

What it is. The full journey from "you press Enter" to "the page appears." Five moves: name → number → pipe → ask → answer (then draw).

📖Jargon decoder
DNS (Domain Name System)
The internet's phone book. It translates a human name like example.com into a machine address (an IP).[AWS]
IP address
A computer's numeric address on the network, e.g. 192.0.2.44 (IPv4, 32-bit) or a longer IPv6 form (128-bit). DNS exists so you don't memorize these.[MDN]
Port
If the IP gets you to the right machine, the port number gets you to the right program on it. Web defaults: 80 for HTTP, 443 for HTTPS (assigned by IANA).[MDN]
TCP / TLS
TCP is the reliable two-way "pipe" the browser opens to the server. TLS is the encryption layer added on top for HTTPS (the padlock).[MDN]

How DNS resolves a name. Your computer asks a recursive resolver (usually run by your ISP). If it hasn't cached the answer, it walks a chain: a root nameserver points it to the TLD server for .com, which points to the domain's authoritative nameserver, which returns the actual IP.[Cloudflare] Each answer is cached for a set time — its TTL — so repeat lookups skip the chain.

🌍Real-world scenario

A user emails you: "your site is down!" — but only for them. You can reach it fine. Knowing this pipeline, you check systematically: Is it DNS (their resolver has a stale or broken record)? The TCP/TLS connection (a certificate or firewall issue)? Or the server's HTTP response (a 500 error)? Each stage in the diagram is a place the request can fail — and a place you can look.

⚠️Watch out (an honest simplification)

The DNS chain shown is the no-cache worst case; in reality your OS and browser also cache, so most lookups short-circuit. And "always TCP" isn't strictly true anymore: modern HTTP/3 runs over QUIC (built on UDP) and merges the connection and encryption handshakes. For a mental model, the five moves above are exactly right.

1.7HTTP up close: methods, status codes, headers — and why it's "stateless"

What it is. HTTP is the request/response language of the web. A request carries a method (the verb), a path, headers, and an optional body. A response carries a status code, headers, and an optional body.[MDN] You'll meet HTTP everywhere — pull these into long-term memory now.

Methods (the verbs) — and two words that matter: safe and idempotent

📖Jargon decoder
Safe
The method is read-only — it isn't meant to change anything on the server. (RFC 9110 §9.2.1)[RFC 9110]
Idempotent
Doing it twice has the same end effect as doing it once — so it's safe to retry after a network hiccup. (RFC 9110 §9.2.2) This is about the server's state, not the response: a second DELETE may return 404 yet the method is still idempotent.[RFC 9110]
MethodTypical useSafe?Idempotent?
GETRead / fetch a resourceYesYes
HEADLike GET but headers only (check existence/size)YesYes
OPTIONSAsk what's allowed (used in CORS preflight)YesYes
POSTCreate / submit / trigger (general write)NoNo
PUTReplace a resource entirelyNoYes
PATCHPartially update a resourceNoNo*
DELETERemove a resourceNoYes

*PATCH is defined in RFC 5789, not RFC 9110, and is generally neither safe nor idempotent (though a specific PATCH can be designed to be).[MDN] All safe methods are idempotent; the reverse isn't true — PUT and DELETE change data yet are idempotent.

Status codes (the answer's headline)

The first digit tells you the category, so you can react sensibly even to a code you don't recognize:[RFC 9110]

Headers (the metadata on every message)

Why HTTP is "stateless" — and why you should love it

MDN puts it bluntly: "HTTP is stateless: there is no link between two requests being successively carried out on the same connection."[MDN] Each request is self-contained; the server keeps no built-in memory of your previous ones.

🔑Key idea — statelessness is a feature

Because no single request depends on the server remembering the last one, any server can handle any request. That's the property that will let us put 50 identical servers behind a load balancer in Section 3 and route traffic to whichever is free. State (your login, your cart) is layered on top using cookies + a shared session store — not baked into the protocol.

🌍Real-world scenario

You log in, and the server sends Set-Cookie: session=abc123. Your browser attaches Cookie: session=abc123 to every later request. That's how a forgetful protocol "remembers" you: the identity rides along on each self-contained request, and the server looks abc123 up in a shared store. Remember this — it's exactly why "sticky sessions" become a problem once we add multiple servers (Section 3).

1.8Functional vs. non-functional requirements

What they are. Two kinds of "what we need." Functional requirements are what the system does; non-functional requirements are how well it must do it.[GeeksforGeeks]

📋 Functional (the features)

  • "Users can post a tweet."
  • "Search returns matching products."
  • "Payments are processed."
  • Answers: what can it do?

⚙️ Non-functional (the qualities)

  • "Feed loads in under 200 ms."
  • "99.99% available."
  • "No accepted order is ever lost."
  • Answers: how well must it behave?

Why it matters. Non-functional requirements (NFRs) usually can't be bolted on later — an availability or scale target shapes the whole architecture. That's why the very first step of system design is dragging these into the open. Here are the NFRs you'll meet constantly, with grounded definitions:

QualityQuestion it answersAnchor
ScalabilityCan it handle more load by adding resources?Horizontal (more machines) vs. vertical (bigger machine).
AvailabilityWhat fraction of the time is it reachable?Measured in "nines": 99.9% ≈ 8.8 hrs down/year; 99.999% ≈ 5 min/year.[AWS]
LatencyHow long does one request take?Report percentiles (p95/p99), not just the average (see 1.9).
DurabilityWill stored data survive (not be lost)?Distinct from availability. Amazon S3 is designed for "eleven nines" (99.999999999%) durability.[AWS]
ReliabilityDoes it keep doing the right thing despite faults?The umbrella property; availability is one part of it.[AWS]
ConsistencyDo all readers see the same, latest data?A spectrum from strong to eventual — the big distributed trade-off (Section 6).
MaintainabilityHow cheaply can people change it later?Modularity, testability, etc. — ISO/IEC 25010's definition.[ISO 25010]
⚠️Watch out — durability ≠ availability

The most-confused pair. Durability = "is my data still there?" Availability = "can I reach the service right now?" A system can be extremely durable (never loses data) yet temporarily unavailable (you can't read it this minute), and vice versa. They're independent dials.[AWS]

1.9Latency vs. throughput — and the numbers worth memorizing

What they are. Latency is how long one operation takes (e.g. 50 ms per request). Throughput is how many you can do per unit time (e.g. 10,000 requests/second). They're different and often in tension: batching work can raise throughput while hurting the latency of an individual item.

🌍Real-world scenario

A supermarket checkout: latency is how long your cart takes to ring up; throughput is how many customers the store clears per hour. Opening more lanes (more servers) raises throughput. Making each scanner faster lowers latency. A manager who only watches throughput might let your personal wait get miserable — which is why we track tail latency (the slow few), not just the average.

Why the numbers matter. Good design comes from an intuition for relative speed: memory is wildly faster than disk, which is faster than a cross-ocean network hop. Jeff Dean's famous "Latency Numbers Every Programmer Should Know" capture the orders of magnitude. The right-hand column scales every figure up by ~1 billion (1 ns → 1 second) to make the gulfs human.

OperationApprox. timeIf 1 ns = 1 second…

These figures come from Jeff Dean's ~2012 talk (popularized via a widely-cited gist).[Jeff Dean gist] Hardware has moved on — modern NVMe SSDs beat the SSD numbers by an order of magnitude. Memorize the relative ladder (cache ≪ RAM ≪ SSD ≪ disk ≪ cross-ocean network), not the exact digits. One number that barely improves: the ~150 ms intercontinental round trip — it's bounded by the speed of light, not technology.[Norvig]

1.10Estimation primer: powers of two, data sizes, and QPS

What it is. "Back-of-the-envelope" estimation — quick, rough math to sanity-check a design before building it. Will this fit in memory? Is this 30 requests/second or 30,000? You don't need precision; you need the right order of magnitude. (We'll do full sizing against real targets in the capstones; this is the toolkit.)

Read powers of two as plain thousands

The single most useful trick: each step of 2¹⁰ (×1,024) is almost exactly ×1,000 in decimal, because 2¹⁰ = 1,024 ≈ 10³. So you can read binary sizes as everyday thousand/million/billion in your head.[ByteByteGo]

PowerExact valueApprox.Unit
2¹⁰1,024~1 thousand1 KB
2²⁰1,048,576~1 million1 MB
2³⁰1,073,741,824~1 billion1 GB
2⁴⁰1,099,511,627,776~1 trillion1 TB
2⁵⁰1,125,899,906,842,624~1 quadrillion1 PB
📖Jargon decoder — KB vs. KiB

Strictly, "kilo" means 1,000 (decimal), so the standards body IEC coined kibibyte (KiB) = 1,024, mebibyte, gibibyte, etc. for the binary values.[Wikipedia] In practice the industry uses KB/MB/GB loosely for the 1,024-based values. For estimation the ~2.4%-per-step error is harmless — but it compounds (2⁵⁰ is ~12.6% above 10¹⁵), which is why a "1 TB" disk (decimal) shows up as only ~931 GiB (binary) in your OS.

Sizes of common things

  • 1 ASCII character = 1 byte. So N characters of plain English text ≈ N bytes.[ByteByteGo]
  • UTF-8 is 1–4 bytes per character — 1 for English, 2–3 for most accented/CJK text, 4 for emoji. Budget 2–4 bytes/char for international text.[Wikipedia]
  • A UUID is 16 bytes stored as raw binary, but 36 bytes as its hyphenated text form — a billion of them is ~16 GB vs ~36 GB.[Wikipedia]
  • A short post (a tweet-sized record with metadata) ≈ a few hundred bytes to ~1 KB.

Turning "requests per day" into QPS

🔑The one constant to memorize

There are 86,400 seconds in a day (24 × 3,600). So average QPS = requests per day ÷ 86,400. A handy sibling: a 30-day month has ~2.6 million seconds, so ~2.6 million requests/month ≈ 1 request/second.[ByteByteGo]

📖Jargon decoder — QPS

QPS = Queries (or requests) Per Second, the standard unit of load. Worked example: 2,500,000 requests/day ÷ 86,400 ≈ 29 QPS on average. Plan for peak QPS (commonly ~2× average) so you have headroom for busy hours.

⚠️Watch out

A day is not a million seconds — it's ~86,400 (≈10⁵). One million seconds is ~11.6 days. Mixing these up overstates capacity by ~11×. When estimating fast, round 86,400 up to 100,000 — it's simpler and conservatively gives a slightly lower QPS.

✅Section 1 recap — your foundation

A server is a program waiting for requests; behind it sit a web server and your application server. Clients and servers talk in request/response round-trips over HTTP, which is stateless — the property that will soon let us run many identical servers. A request travels name → number → pipe → ask → answer (DNS → IP/port → TCP/TLS → HTTP). Your code runs as a process, inside a container, possibly inside a VM/instance, on a machine. And you can now reason in orders of magnitude — latency ladders, powers of two, and QPS. Section 3 cashes all of this in the moment one server isn't enough.

Section 2 · Foundational

Networking & Cloud Topology: How a Request Travels

In Section 1 you followed a request from the browser to a server. Now we go one level deeper: the pipes a request travels through (TCP, UDP, TLS), why reusing those pipes matters, the proxies that sit in the middle, and the physical map of the cloud — regions, zones, and edges — that decides how fast and how reliable your system can ever be.

2.1TCP vs. UDP, and the HTTPS handshakes (the "setup cost")

What they are. Below HTTP sit two ways to actually move bytes across the network. TCP is the careful courier; UDP is fire-and-forget.

📖Jargon decoder
TCP (Transmission Control Protocol)
Connection-oriented, reliable, ordered. It guarantees your bytes arrive, and in order, retransmitting anything lost. The default for web pages, APIs, email, file transfer.[MDN]
UDP (User Datagram Protocol)
Connectionless, no guarantees, low overhead. A sender just shoots packets; some may be lost, reordered, or duplicated. The payoff is speed.[MDN]
Round trip (RTT)
One message out and its reply back. Handshakes are measured in round trips — and each one is time the user waits before seeing anything.

How a secure connection is set up. Recall the request lifecycle from §1.6. For HTTPS, before the first HTTP byte flows, three things happen in order — and each costs round trips:

  1. DNS lookup — name → IP (§1.6).
  2. TCP three-way handshake — SYN → SYN-ACK → ACK. The two sides agree to talk before any data moves.[MDN]
  3. TLS handshake — authenticates the server (via its certificate), negotiates keys, and switches on encryption.[RFC 8446]
🔑Key idea — TLS got faster

The encryption handshake used to be expensive. TLS 1.2 needed two extra round trips; TLS 1.3 cut that to one (1-RTT), and for a resumed connection, 0-RTT sends encrypted data in the very first message.[Cloudflare] (0-RTT trades away some security, so it's reserved for safe, non-state-changing requests.)

✓ Use TCP when

  • Every byte must arrive correctly and in order: web pages, APIs, payments, file transfer.
  • You can tolerate the handshake + retransmit cost for reliability.

✓ Use UDP when

  • Late data is useless: live voice/video, gaming, telemetry.
  • Small, fast, fire-and-forget exchanges: DNS queries.
  • You'll handle (or happily skip) loss in the app.
🌍Real-world scenario

On a video call, a packet that arrives 300 ms late is worse than useless — you'd rather drop it and stay in sync, so the call uses UDP. But when you then send a bank transfer in another tab, losing or reordering bytes is unacceptable, so that uses TCP. Same device, same network, two protocols — chosen by what failure you can tolerate.[MDN]

2.2Connection reuse: keep-alive, HTTP/2, and HTTP/3

Why it exists. You just saw the setup cost. Now imagine paying it for every image, script, and stylesheet on a page — dozens of handshakes. The history of HTTP is largely the story of avoiding that.

📖Jargon decoder
Persistent connection / keep-alive
Keep one TCP connection open and reuse it for many requests, so you pay the handshake once. The default since HTTP/1.1.[MDN]
Multiplexing
Running many independent request/response exchanges (called streams) at the same time over a single connection. HTTP/2's headline feature.[RFC 9113]
Head-of-line (HoL) blocking
When one stuck item holds up everything queued behind it — like one slow shopper freezing a checkout lane.

How it evolved. HTTP/1.0 opened a fresh connection per request and closed it after — pay the handshake every time. HTTP/1.1 added persistent connections (reuse) and pipelining, but pipelining suffered HoL blocking and is effectively unused today.[MDN] HTTP/2 multiplexes many streams over one connection and compresses headers (HPACK), so browsers no longer open ~6 parallel connections per origin.[RFC 9113]

⚠️Watch out — two things people get wrong

HTTP/2 server push is deprecated (Chrome disabled it by default in version 106, 2022, after ~1% adoption); use 103 Early Hints / preloading instead.[Chrome] And HTTP/2 fixes head-of-line blocking only at the HTTP layer — because all streams share one TCP connection, a single lost packet still stalls every stream (TCP-level HoL blocking). HTTP/3 moves onto QUIC (over UDP), tracking loss per-stream and merging the transport+TLS setup into a single round trip.[Cloudflare]

🌍Real-world scenario

A product page pulls 120 small assets. On HTTP/1.1 the browser opens ~6 connections and queues the rest, so the page paints in waves. Flip the server to HTTP/2 and all 120 stream over one warm connection in parallel — often a visible speed-up with zero code change. This is "free performance" you get just by serving modern HTTP.

2.3Forward proxy vs. reverse proxy

What they are. A proxy is a middleman that forwards requests on someone's behalf. The direction it faces is the whole distinction. (You already met the reverse proxy in §1.3 — here's its mirror image.)

📖Jargon decoder
Forward proxy
Sits next to the clients and represents them to the outside world — it hides who is asking. The client is configured to route through it. Used for corporate egress control, content filtering, and caching outbound traffic.[Apache]
Reverse proxy
Sits in front of the servers and represents them to clients — it hides the backend. No client configuration needed; the client thinks the proxy is the website. Used for TLS termination, load balancing, caching, and security.[F5/NGINX]
🌍Real-world scenario

At a company, all employee web traffic routes through a forward proxy that blocks malware domains and logs usage — it speaks for the clients. Meanwhile, that company's public website sits behind a reverse proxy (Nginx) that terminates HTTPS and spreads requests across app servers — it speaks for the servers. The same software, pointed in opposite directions.

2.4Cloud topology: regions, availability zones, and edges

What it is. The cloud isn't an abstract "somewhere." It's a physical hierarchy of buildings, and where you place your system in that hierarchy decides how much failure it can survive and how close it sits to users.

📖Jargon decoder (AWS terms; other clouds use near-identical concepts)
Region
A separate geographic area (e.g. us-east-1, N. Virginia), deliberately isolated from other regions for fault tolerance. What you create in one region lives only there.[AWS]
Availability Zone (AZ)
"One or more discrete data centers with redundant power, networking, and connectivity" within a region. AWS guarantees at least three AZs per region, physically separated (within ~100 km) yet linked with single-digit-millisecond latency.[AWS]
Edge location / Point of Presence (PoP)
Hundreds of small sites near end users where a CDN caches content. They don't run your app — they shorten the distance to your users.[AWS CloudFront]
Failure domain
The blast radius of one failure. An AZ is designed to be a single failure domain — assume it can all go down at once.

✓ Climb the reliability ladder…

  • Single AZ: simplest, cheapest — but one data-center fault takes you down.
  • Multi-AZ (one region): survive losing a whole data center, still low latency. The standard for production.
  • Multi-region: survive a whole region outage.

✓ …paying as you climb

  • Each step adds cost and complexity.
  • Multi-region means thousands of km between copies → higher latency and usually asynchronous replication with harder consistency trade-offs (Section 6).
  • Match the rung to your actual reliability requirement.[Microsoft]
🌍Real-world scenario

You launch in a single AZ. One night that data center loses power and your whole app goes dark. You redeploy across three AZs in the same region behind a load balancer — now a zone can fail and the other two carry the traffic, with replication between them fast enough to stay in sync. Later, to serve users in Asia, you don't move the region — you put edge/CDN PoPs near them so images and cached pages load locally.

2.5The speed-of-light latency floor

What it is. There's a hard limit to how fast any request can travel, set by physics. No hardware, protocol, or budget can beat it — the only lever is distance.

How it works. Light in a vacuum travels ~300,000 km/s. In optical fiber it's slower — about two-thirds of that (~200,000 km/s) because glass has a refractive index near 1.5 — roughly 5 microseconds per kilometer.[High Performance Browser Networking] Because this is propagation delay, it sets a floor you can compute from a map:

HopDistancePropagation floor (round trip)
Within one data centermeterswell under 1 ms
Across AZs in a regionup to ~100 kmsingle-digit ms[AWS]
New York ↔ London~5,585 km~56 ms (physics floor)[HPBN]
⚠️Watch out — floor vs. reality

That ~56 ms NY↔London is the theoretical propagation floor. Real-world round trips are higher (often 80–150 ms) because of indirect cable routing, router hops, and queuing — those are extra delays on top of the floor, not the floor itself. The takeaway holds: cross-ocean latency barely improves over the years because the dominant term is distance ÷ speed of light.

🔑Key idea

You can't make light faster — so you shorten the path. This is the physics-level reason CDNs/edge PoPs, multi-region deployments, and "put data near the user" exist. It's also why AZs are kept within ~100 km: close enough for fast synchronous replication, far enough to fail independently. This single trade-off — distance vs. failure isolation — echoes through the rest of the guide.

Section 3 · The motivating spine

Scaling Out: Load Balancing & Statelessness

This is the section the whole field is built around. One server is never enough — it's slow under load and it's a single thing that can break. The fix (more servers) immediately creates a new problem (who routes to them? and why do users get logged out?). Load balancers and statelessness are the answers, and almost everything later builds on them.

3.1Why one server isn't enough — and two ways to grow

The problem. If your whole app runs on one machine, that machine is a single point of failure — and your ceiling.

📖Jargon decoder
Single point of failure (SPOF)
Any one component whose failure stops the entire system, because there's no backup. One server is a SPOF: a hardware fault, a crash, a bad deploy, or simple overload takes everything down with it.[def.]
Vertical scaling (scale up)
Make the one machine bigger — more CPU/RAM.[Google Cloud]
Horizontal scaling (scale out)
Add more machines running the same program.[AWS]
High availability (HA)
Staying up despite failures — achieved by removing SPOFs through redundancy (multiple interchangeable machines).

✓ Vertical scaling — good when

  • You need a quick win with no code changes (the app doesn't know).
  • The workload is hard to distribute (e.g. a single big database — for now).
  • Limits: hits the largest-instance ceiling, usually needs a reboot to resize, and stays a SPOF.[Google Cloud]

✓ Horizontal scaling — good when

  • You need to grow past one machine and want redundancy (lose one, others carry on).
  • You can run many interchangeable copies.
  • Cost: needs a load balancer, requires statelessness (3.4), and adds distributed-systems complexity.
🌍Real-world scenario — the classic "10× traffic spike"

Your app runs happily on one server. You get featured in the press and traffic jumps 10×. Scaling up buys a night's sleep, but you quickly hit the biggest instance available — and you're still one power-supply away from a total outage. So you switch to scaling out: run several identical app servers and put a load balancer in front to spread the flood across them. More capacity and no single point of failure — the two payoffs arrive together. That load balancer is exactly what the next topic is about.

⚠️Watch out — redundancy isn't free

AWS guidance is blunt: each time you remove a SPOF, cost and complexity at least double — and sometimes accepting a SPOF is the right economic call. "Make everything redundant" is not automatically correct; match it to what an outage would actually cost you.[AWS]

3.2Load balancers: the traffic cop in front of your servers

What it is. A load balancer (LB) is the single entry point clients hit; it distributes incoming requests across a pool of backend servers, and routes only to healthy ones. That buys two things at once: more throughput (servers share the work) and higher availability (route around failures).[AWS ELB]

Layer 4 vs. Layer 7 — how much the LB "reads"

How it picks a server (the algorithms)

⚠️Watch out — don't move the SPOF

If you put one load balancer in front of many servers, you've just made the LB the new single point of failure. Real setups run the LB itself redundantly — multiple LB nodes across AZs with DNS failover, or a managed LB (AWS ELB, Cloudflare) that's internally redundant by design. That's why beginners are told to use a managed load balancer rather than hand-running one box.[AWS]

🌍Real-world scenario

Your API and your image service live on different server pools. An L7 load balancer reads the URL and sends /api/* to the API servers and /images/* to the image servers — one front door, content-aware routing. For a high-throughput game backend that just needs raw TCP spread with static IPs, an L4 balancer is the faster, simpler fit. And for a fleet of cache servers, consistent hashing keeps each key on the same cache even as you add nodes — so you don't blow away the whole cache on a deploy.

3.3Health checks & failover: detecting and routing around the dead

What it is. A load balancer only sends traffic to healthy servers — but how does it know which are healthy? It health-checks them: pings a status endpoint on a fixed interval and counts consecutive passes/fails.

How it works. A target flips to unhealthy only after several consecutive failures (so one blip doesn't eject it) and back to healthy after several consecutive successes (so it must prove stable recovery). AWS ALB defaults: check every 30 s, out after 2 failures, back after 5 successes.[AWS]

📖Jargon decoder — Kubernetes probes (the same idea, named precisely)
Liveness probe
"Is it alive?" If it fails, the container is restarted — catches a process that's stuck (e.g. deadlocked) but technically still running.[Kubernetes]
Readiness probe
"Can it serve traffic right now?" If it fails, the instance is removed from rotation (no restart) — for something alive but warming up or waiting on a dependency. Recovers → added back.[Kubernetes]
Startup probe
Gives a slow-booting app a grace period before liveness/readiness kick in, so it isn't killed mid-boot.
🌍Real-world scenario

You deploy a bad build to one of three servers and it starts returning errors. Its health check fails twice; the load balancer quietly takes it out of rotation and users keep hitting the two healthy servers — most never notice. This is also how rolling deployments work safely: update one server at a time, let it pass its health check, then move to the next.

3.4The bug scaling created: statelessness & sessions

Adding servers fixed capacity and availability — but it quietly broke logins. Here's the single most important habit in scaling, told as the bug it fixes.

📖Jargon decoder
Stateful service
The server keeps client-session data (login, shopping cart) in its own memory between requests — which ties that user to that one server.[Redis]
Stateless service
The server keeps no per-client memory between requests, so any instance can handle any request. (This is HTTP's own statelessness from §1.7, applied to your app.)

The bug. Put session data in one server's memory, then add a load balancer, and consecutive requests from the same user can land on different servers. The second server has never seen this user.

Two fixes. One patches around the symptom; one removes the cause.

① Sticky sessions (a patch)

  • The LB pins each user to the same server (via a cookie or IP hash) so their in-memory state stays reachable.[NGINX]
  • Downsides: load gets uneven, and if that server dies the user's state dies with it — they're logged out anyway.

② Externalize the session (the real fix)

  • Move session data out of any one server into a shared store (e.g. Redis) every server can read.
  • Now servers are truly stateless & interchangeable — any one can serve any request, and losing a server loses nothing.[Redis]
🔑Key idea — this is why §1.7 mattered

Statelessness is the precondition for horizontal scaling. Because no request depends on a server remembering the last one, you can add, remove, or lose servers freely — which is exactly what load balancing, autoscaling (Section 11), and "lose one, the rest carry on" all rely on. Keep your app servers stateless and push state to shared data stores. That single habit unlocks most of what follows.[Google Cloud]

🌍Real-world scenario

An e-commerce site uses sticky sessions and it "works" — until a Black Friday traffic spike triggers autoscaling, which removes a server when load drops. Every shopper pinned to that server loses their cart mid-checkout. The fix: move carts/sessions to Redis. Now servers come and go freely and no cart is ever tied to a machine's lifespan.

✅Recap — Sections 2 & 3

A request rides TCP (reliable) or UDP (fast), pays a one-time handshake setup cost you amortize with keep-alive / HTTP/2 multiplexing, and may pass through a forward or reverse proxy. Physically it lives in a hierarchy of regions → AZs → edges, all bounded by the speed-of-light floor (so you move data closer to users). When one server isn't enough, you scale out behind a load balancer (L4/L7, picking servers by round-robin/least-connections/consistent-hashing), which uses health checks to route around failures — and you keep servers stateless (sessions in a shared store) so any server can serve any request. Next, in the data layer, that "shared store" grows up into the real database.

Section 4 · Foundational

The Data Layer: Storage, Durability, Replication & Sharding

In Section 3 we made app servers stateless and pushed their state into a "shared store." This section is about that store growing up into a real database — the single most precious, most contended component in your whole system. We'll make it correct (SQL & ACID), fast (indexes, caching the right things), durable (write-ahead logging), and finally scalable (replicas and shards) — in that order, because each fix is the answer to the previous one's limit.

4.1The database: the shared source of truth

What it is. A database is the durable, authoritative home for your data — the "shared store" from §3, made real. Every stateless app server reads from and writes to it, so they all agree on one copy of the truth, and that copy survives restarts (unlike a server's RAM).

🔑Key idea — and the tension for the rest of this section

Making the database the single source of truth is what lets you scale app servers freely. But it also makes the database the most contended and most critical component — everything funnels into it. The rest of this section is the toolkit for keeping that one thing correct, durable, fast, and (eventually) bigger than one machine.

4.2Relational databases (SQL) and ACID

What it is. The relational model stores data in tables — rows (records) and columns (fields) with a defined schema — and links tables together with keys. You query it with SQL.[PostgreSQL]

📖Jargon decoder
Schema
The fixed structure of a table — which columns exist and their data types. Enforced by the database.
Primary key
A column whose value uniquely identifies each row (unique, never empty). E.g. user_id.[PostgreSQL]
Foreign key
A column that references another table's primary key, linking the two and enforcing "referential integrity" — you can't reference a row that doesn't exist.[PostgreSQL]
SQL
A declarative language: you describe what data you want; the database figures out how to get it.
Transaction
A bundle of steps treated as one all-or-nothing unit (BEGIN … COMMIT, or ROLLBACK to undo).[PostgreSQL]

Why it exists. Structure + integrity + powerful ad-hoc queries. The headline guarantee is ACID — the four properties that make transactions trustworthy:

LetterMeansIn plain terms
A — AtomicityAll-or-nothingEvery step in the transaction happens, or none does. No half-finished results.
C — ConsistencyValid → validA transaction moves the DB from one valid state to another, never violating its rules/constraints.
I — IsolationNo interferenceConcurrent transactions don't see each other's half-done work. (Tunable via isolation levels.)
D — DurabilityCommitted = permanentOnce committed, it survives crashes and power loss (the mechanism is §4.5).

ACID definitions per the MySQL glossary; isolation levels (Read Uncommitted → Serializable, default Read Committed) per PostgreSQL.[MySQL] [PostgreSQL]

🌍Real-world scenario — the $100 transfer

Move $100 from Alice to Bob: debit Alice and credit Bob are two writes that must both happen or neither — otherwise money is created or destroyed. Wrap them in one transaction and atomicity guarantees it's all-or-nothing, even if the server crashes in between. This is the canonical reason ACID databases run the world's money.[PostgreSQL]

4.3NoSQL families — and choosing SQL vs. NoSQL

What it is. "NoSQL" is an umbrella for non-relational, purpose-built databases that drop the rigid table model for something tuned to a specific shape of data and access pattern.[AWS] The four you'll meet most:

FamilyStores data asCanonical exampleGreat for
Key-valueA key → an opaque value, looked up only by keyRedis, DynamoDBCaches, sessions, simple fast lookups[Redis]
DocumentNested JSON-like documents; no enforced schemaMongoDBFlexible, evolving records (catalogs, profiles)[MongoDB]
Wide-columnRows with a flexible, sparse set of columnsCassandraMassive write-heavy, time-series data[Cassandra]
GraphNodes + relationships (edges) with propertiesNeo4jHighly connected data (social graphs, fraud)[Neo4j]

What NoSQL trades. Generally: flexible/loose schemas and easy horizontal scaling, in exchange for fewer cross-table joins and (often) weaker default consistency — you model the data around the exact queries you'll run.[AWS]

⚠️Watch out — "NoSQL = no ACID/no joins" is outdated

NoSQL is a heterogeneous bucket — don't over-generalize. Several once-true clichés no longer hold: DynamoDB and MongoDB both support multi-item ACID transactions today. Treat the trade-offs above as tendencies, not laws, and check the specific database.

✓ Reach for relational (SQL) when

  • Data is structured and highly related; you need strong consistency and integrity.
  • You'll run varied, ad-hoc queries and joins.
  • It's the safe default — start here unless you have a reason not to.[AWS]

✓ Reach for NoSQL when

  • You have one dominant, well-known access pattern and need it at huge scale.
  • The data is naturally key-value, document, graph, or sparse/columnar.
  • Choose by access pattern, not by "newer is better."
🌍Real-world scenario

One app uses several stores by job: PostgreSQL for orders and payments (relational + ACID), Redis (key-value) for sessions and caching, MongoDB (document) for a product catalog whose fields vary by category, and a graph database for "people you may know." Picking by access pattern beats forcing everything into one model — this is "polyglot persistence."

4.4Indexes, query plans, and normalization

What an index is. A separate, sorted data structure (by default a B-tree) that lets the database jump to matching rows instead of scanning the whole table. Without it, finding rows means reading every row — a sequential (full) scan.[PostgreSQL]

⚠️Watch out — indexes aren't free

Every index speeds up reads but slows down writes (each insert/update must also update the indexes) and costs storage. "The number of indexes is a multiplier for the cost of an insert." So index the columns you filter/sort by — not every column.[Use-The-Index-Luke] [PostgreSQL]

Reading the plan. Databases pick a strategy per query; EXPLAIN shows whether it's a sequential scan (slow on big tables) or an index scan. When a query is mysteriously slow, this is the first place to look.[PostgreSQL]

Normalization vs. denormalization

Normalization (store once)

  • Split data into tables so each fact lives in one place, linked by foreign keys.
  • No duplication, easy correct writes — but reads may need joins to recombine.[Built In]

Denormalization (duplicate for speed)

  • Deliberately duplicate/precombine data to avoid expensive joins → faster reads.
  • Cost: more storage and write complexity — you must keep every copy in sync.
🌍Real-world scenario

A WHERE email = … query crawls because it scans all 10 million users. You add an index on email and it returns instantly. Later, a report joins five normalized tables and is too slow for a dashboard — so you denormalize a precomputed summary table that's fast to read, accepting the extra work to keep it updated. Indexing and (de)normalizing are the two main read-speed levers before you reach for replicas or sharding.

4.5How "committed" becomes permanent: WAL, fsync & backups

What it is. The "D" in ACID isn't magic — it's the write-ahead log (WAL). The rule: changes are written to an append-only log on disk and flushed before they're applied to the actual table files.[PostgreSQL]

📖Jargon decoder
Write-ahead log (WAL)
An append-only record of every change, written to disk first. After a crash, the DB replays the log to recover any changes not yet applied (called "REDO" / roll-forward).[PostgreSQL]
fsync
A system call that forces buffered writes past the OS and disk caches onto the physical platter/flash — so they truly survive a power cut. Turning it off risks "unrecoverable data corruption."[PostgreSQL]
Commit-then-ack
The DB returns "success" to the client only after the transaction's WAL records are durably on disk.
🌍Real-world scenario

A user taps "Pay" and sees Payment successful — then the data-center loses power a millisecond later. Because the payment's WAL record was fsync'd to disk before that success message was sent, the database replays it on restart and the payment is intact. The same WAL also powers point-in-time recovery: a base backup plus the archived log lets you restore the database to any moment — say, just before a bad migration. (And: test your restores — a backup you've never restored is a guess.)[PostgreSQL PITR]

4.6Connection pooling and the single-database bottleneck

The problem. Every database connection costs memory and a backend process, so databases cap how many you can have at once — PostgreSQL's default max_connections is 100.[PostgreSQL] But horizontal scaling gave you many stateless app instances, each opening several connections — and autoscaling can multiply them in a "connection storm" that blows past the limit.

📖Jargon decoder — connection pooler

A connection pooler (PgBouncer, AWS RDS Proxy) sits between your apps and the database and multiplexes many client connections onto a small set of real DB connections — handing a connection back to the pool when a client is done.[PgBouncer] [AWS RDS Proxy]

🔑Key idea — the database is the bottleneck

You can add app servers all day, but they all funnel into one primary database. That single writer is now your scaling ceiling. The good news: web workloads are overwhelmingly read-heavy — reads vastly outnumber writes — which points straight at the next two tools: copy the data to more readers (replication, §4.8) and cache the hot reads (Section 5).

4.7Right tool for the data: object storage & search indexes

What it is. Not everything belongs in the relational database. Two kinds of data get their own purpose-built homes, with the database keeping only a reference.

📖Jargon decoder
Object (blob) storage — e.g. Amazon S3
Stores large unstructured files (images, video, backups) as objects in buckets, each addressed by a key and fetched over HTTP. Built for near-unlimited scale and ~eleven nines (99.999999999%) of durability.[AWS S3]
Inverted index — e.g. Elasticsearch / OpenSearch
A structure that maps each word → the documents containing it, enabling fast, relevance-ranked (BM25) full-text search.[Elastic]

Why files don't go in the DB

  • Large binaries bloat the database and its backups and waste its cache.
  • Store the file in object storage; keep only its URL + metadata in the DB.[Microsoft Research]

Why search gets its own engine

  • A leading-wildcard LIKE '%term%' can't use a normal B-tree index and has no relevance ranking — slow and dumb.
  • Index your text into a search engine and query that instead.[Elastic]
🌍Real-world scenario

A user uploads a 50 MB video. You store the file in S3 and save just its URL in the orders table; the page (and your CDN, Section 5) serves the file straight from S3. Separately, to power the product search box, you index product names and descriptions into Elasticsearch — so a search returns ranked results in milliseconds instead of LIKE-scanning the primary database.

4.8Replication and read replicas

What it is. Replication keeps copies of the same data on multiple servers. The common shape is leader–follower (a.k.a. primary–replica): one leader takes all writes and streams its WAL to one or more followers, which can serve reads. Since web traffic is read-heavy, this offloads the leader and scales reads.[PostgreSQL] [AWS RDS]

Asynchronous (the common default)

  • Leader acks the commit immediately, ships changes to followers after.
  • Low write latency — but followers lag, and a failover can lose the last few writes.[PostgreSQL]

Synchronous

  • Leader waits for a follower to confirm before acking the commit.
  • Stronger durability/consistency — but higher write latency.[PostgreSQL]

The catch: replication lag. Followers trail the leader by some delay — and that creates a subtle, famous bug.

🌍Real-world scenario

You change your display name and the page reloads — still showing the old name, because the reload hit a replica 200 ms behind the leader. Users read this as "my change was lost." Common fix: for a short window after a write, route that user's reads to the leader (read-your-writes), or wait for the replica to catch up. We'll meet this stale-data problem again, named and generalized, in Section 6 (consistency).

4.9Sharding: splitting the data across machines

What it is. Replicas scale reads, but every write still hits one leader, and all the data still has to fit on one machine. Sharding (horizontal partitioning) breaks that ceiling: split the dataset so each shard holds a different subset of the rows, on its own server.[MongoDB]

📖Jargon decoder — sharding vs. replication

Replication copies the same data to many nodes (for read scaling + availability). Sharding puts different data on different nodes (for write/storage scaling). They're complementary — each shard is usually itself replicated. The shard key is the column whose value decides which shard a row lives on.[MongoDB]

StrategyHow keys map to shardsTrade-off
RangeContiguous key ranges per shard (A–H, I–P…)Great for range queries; risks hotspots on sequential keys[MongoDB]
HashHash the key to spread evenlyEven load; range queries become cluster-wide broadcasts[MongoDB]
DirectoryA lookup table maps keys → shardsFlexible; the lookup service is one more thing to run[Vitess]

Consistent hashing (from §3.2) is the refinement that keeps reshuffling small when you add or remove a shard.

⚠️Watch out — sharding is the costliest tool

It brings real pain: hot shards — the "celebrity problem," where one key (a celebrity's posts, or a monotonically increasing timestamp) concentrates load on a single shard; cross-shard queries and joins that must fan out to every shard; and resharding, which is a hard, multi-stage operation.[MongoDB] [Vitess] Shard last, only after replicas, caching, and a bigger box aren't enough — and choose the shard key very carefully.

🌍Real-world scenario

Your posts table grows past what one machine can store or write. You shard by user_id so each user's posts live on one shard. But shard by timestamp instead and every new post lands on the "latest" shard — a write hotspot that defeats the whole point. And when a celebrity joins, their shard melts under load. The shard key is the design decision.

✅Recap — Section 4

The database is the shared source of truth. Relational + ACID makes it correct; indexes and (de)normalization make reads fast; the write-ahead log makes "committed" survive crashes. One primary is a bottleneck, eased by connection pooling, then by pushing the right data to object storage and search engines, then by replicating reads across followers (mind the lag), and finally — as a last resort — sharding writes across machines. Notice the recurring theme: reads vastly outnumber writes. That single fact is the doorway to Section 5 — caching the hot reads so they never touch the database at all.

Section 5 · Performance

Making It Fast: Caching & CDNs

Section 4 ended on one fact: reads vastly outnumber writes, and the single database is the bottleneck everything funnels into. Caching is the answer — keep the hot answers in fast memory so most reads never touch the database at all. Then CDNs push that idea all the way out to the user's doorstep.

5.1Why cache? Hits, misses, and the read-heavy win

What it is. A cache stores the result of an expensive fetch (a DB query, a computed page) in faster storage — usually RAM — so that repeat requests are served instantly without redoing the work.[AWS]

📖Jargon decoder
Cache hit
The data is in the cache (and not expired) — return it directly. Fast.
Cache miss
It's absent or expired — fetch from the database, then put it in the cache for next time. AWS notes a miss costs three round trips (cache → DB → write-back) — the "miss penalty."[AWS]
Hit ratio
hits ÷ total requests. A high hit ratio means most requests skip the slow backend — the whole point.

Why it works. RAM is roughly a thousand times faster to read than disk (recall the latency ladder in §1.9), so an in-memory cache like Redis answers in sub-milliseconds.[Redis] Because web traffic is read-heavy, caching the read results is a huge win — AWS reports pairing a cache with a database can give up to ~80× faster reads.[AWS]

🌍Real-world scenario

A product page is viewed a million times a day but its data changes only when the seller edits it. Without a cache, that's a million identical database queries. Cache the rendered result with a short TTL and ~99% of those views are served from RAM in under a millisecond — the database barely notices. That's the read-heavy win in one example.

5.2Where caches live: the cache stack

What it is. "The cache" isn't one place — caching happens at every layer between the user and your database. Each tier is closer to the user (faster) but smaller and less shared.

📖Jargon decoder
Private vs. shared cache
A private cache serves one user (the browser); a shared cache (CDN, reverse proxy) serves many.[MDN]
In-process vs. distributed cache
In-process = data in one app server's own RAM (fastest, but not shared across servers and lost on restart). Distributed = a separate cache server (Redis/Memcached) every app server shares over the network.
Redis vs. Memcached
Both are in-memory key-value stores used as distributed caches; Redis adds richer data types and persistence options, Memcached is a leaner pure cache.

5.3Caching strategies: who writes to the cache, and when

What they are. Four standard patterns for keeping a cache and database in sync. They differ in who loads the cache and when writes propagate.

StrategyHow it worksTrade-off
Cache-aside (lazy loading)App checks cache; on a miss it reads the DB and populates the cache itselfOnly caches what's actually requested; simple. But the first read is a miss, and the cache can go stale.[AWS]
Read-throughThe cache itself loads from the DB on a miss (app only talks to the cache)Simpler app code; the cache library owns the DB lookup.[Microsoft]
Write-throughEvery write updates cache and DB synchronouslyCache always fresh; writes are slower (two writes every time).[Redis]
Write-back (write-behind)Write to cache now; flush to DB asynchronously laterVery fast writes; risk of data loss if the cache dies before flushing.[Redis]
🔑Key idea

Cache-aside is the default for read-heavy apps. But it makes no consistency guarantee — an update straight to the database is invisible to the cache until the cached copy is reloaded or expires. The usual safety net is a TTL so stale data can only live so long.[Microsoft]

5.4Eviction & invalidation: the cache is small, and truth changes

The problem. A cache has a fixed memory budget (Redis's maxmemory), so when it fills up it must evict something. And the source of truth keeps changing, so cached copies go stale.[Redis]

📖Jargon decoder — eviction policies

When full, which key gets thrown out? The three ideas that matter:

LRU — Least Recently Used
Evict whatever hasn't been touched for the longest time. The common default.
LFU — Least Frequently Used
Evict whatever is accessed least often (with decay so old favorites fade).
TTL — Time To Live
A per-key expiry timer; the key auto-deletes when it elapses (independent of memory pressure).

Redis offers a range of policies built on these (plus random and noeviction), and added newer "least-recently-modified" variants in Redis 8.6 — but LRU, LFU, and TTL are the concepts to know.[Redis]

⚠️Watch out — invalidation is genuinely hard

As the saying (attributed to Phil Karlton) goes: "There are only two hard things in Computer Science: cache invalidation and naming things."[Fowler] Knowing when to remove or refresh a cached value — the moment the underlying data changes, no sooner, no later — is one of the most error-prone parts of system design. The pragmatic tools are TTLs (bounded staleness) and explicit invalidation on write.

5.5Cache stampede & hot keys

What it is. A cache stampede (a.k.a. thundering herd / dog-piling): when a popular cached key expires, many in-flight requests all miss at the same instant and stampede the database to recompute the same value at once — which can overload it and keep it in continuous collapse.[Wikipedia]

✓ Taming the stampede

  • Locking / single-flight: only one request recomputes; the rest wait for it.
  • Jittered TTLs: add randomness to expiry so keys don't all expire together.
  • Early / background recompute: refresh just before expiry, off the request path.[Wikipedia]

✓ Hot keys

  • A single key getting a disproportionate share of traffic overwhelms the one cache node that owns it.
  • Mitigate by replicating that key across nodes or splitting it into several physical keys.[Redis]
🌍Real-world scenario

Your homepage is cached with a 5-minute TTL. At peak traffic the key expires and, in the same second, 50,000 requests all miss and slam the database with the identical expensive query — it falls over, and every refresh just re-triggers the pile-up. The fix: a single-flight lock so one request rebuilds the page while the others wait, plus a little TTL jitter so this never lines up so perfectly again.

5.6CDNs: caching at the edge, next to the user

What it is. A CDN (Content Delivery Network) is a geographically distributed fleet of edge servers (the points of presence from §2.4) that cache your content close to users — cutting latency and offloading your origin.[Cloudflare] It's the same caching idea, pushed to the network edge.

📖Jargon decoder
Origin
The authoritative source of the content — often object storage (an S3 bucket, §4.7) or your own server.[AWS CloudFront]
Pull vs. push
Pull (most common): the CDN fetches from your origin on the first request and caches it. Push: you upload content to the CDN ahead of time.
Edge TTL via Cache-Control
How long the edge keeps a copy is set by the HTTP Cache-Control header (§1.7) — the origin tells the CDN how long it may serve the cached version.
🌍Real-world scenario

Your app's images, CSS, JavaScript, and videos are static — identical for every user — so you serve them from a CDN. A user in Tokyo gets them from a Tokyo edge node in a few milliseconds instead of crossing an ocean to your US origin (remember the ~150 ms physics floor from §2.5). Dynamic, per-user content (their account page) is harder to cache — short TTLs, edge compute, or simply not cached.[Cloudflare]

5.7Measuring speed: tail latency (p95/p99), not the average

What it is. The right way to report latency is as percentiles, not an average — because the average hides the slow outliers that users actually feel.[Google SRE]

PercentileMeans
p50 (median)Half of requests are faster than this — the "typical" experience.
p9595% are faster; the slowest 1-in-20 are worse.
p9999% are faster; the slowest 1-in-100. A good rule of thumb: keep p99 within ~3–5× the median.[Google SRE]
p99.9The slowest 1-in-1000 — the deep tail, felt by your heaviest users.
⚠️Watch out — "the tail at scale"

Tail latency matters more than it looks. If a page fans out to 100 backend services in parallel and each is slow (over its p99) just 1% of the time, the page is slow about 63% of the time — because it only takes one slow component (1 − 0.99¹⁰⁰ ≈ 0.63).[Dean & Barroso] So a backend's p99 effectively becomes the frontend's median. Optimize the tail, not the average.

✅Section 5 recap

Cache the hot, read-heavy answers in RAM so they skip the database (hit vs. miss). Caches live in a stack — browser → CDN → in-process → distributed (Redis) → DB. Pick a strategy (cache-aside is the default) and accept that you must handle eviction (LRU/LFU/TTL) and the hard problem of invalidation. Beware the stampede when hot keys expire. CDNs push caching to the edge, next to users. And measure with percentiles (p95/p99) because the tail is what users feel. But caching also just introduced a second copy of your data that can disagree with the source — which is the doorway to Section 6: what "consistency" even means when data lives in more than one place.

Section 6 · Distributed theory

Distributed Reality: Clocks, Consistency, CAP & PACELC

You've now copied data in three ways — replicas (§4), caches (§5), and soon more. The moment data lives in more than one place, those copies can disagree, lag, and conflict. This section is the theory that names those problems honestly: why you can't trust clocks, what "consistency" really means, and the famous CAP and PACELC trade-offs. It's the most abstract section — but every idea here is the generalized form of a stale read you've already met.

6.1Partial failure and the fallacies of distributed computing

What it is. A distributed system is many computers cooperating to look like one. Its defining trait is partial failure: parts break in unpredictable ways while others keep working — and crucially, when a remote node doesn't answer, you can't tell whether it crashed, the request was lost, the reply was lost, or it's just slow.[DDIA Ch.8]

📖The eight fallacies of distributed computing

False assumptions newcomers make (L. Peter Deutsch & James Gosling, Sun Microsystems). Every one of them will eventually bite you:[Wikipedia]

  1. The network is reliable
  2. Latency is zero
  3. Bandwidth is infinite
  4. The network is secure
  5. Topology doesn't change
  6. There is one administrator
  7. Transport cost is zero
  8. The network is homogeneous
🌍Real-world scenario

Your service calls a payment service and gets… silence. Did the charge go through and the reply got lost? Did the request never arrive? Is it just slow and the answer is coming? You genuinely cannot know — so you must design for it (timeouts, retries, idempotency — all of Section 8). This single uncertainty, "the dreaded unknown," is why distributed systems are hard, and it's the thread that runs through the rest of the guide.

6.2You can't trust clocks

What it is. Each machine has its own clock, and they don't agree. Worse, the time-of-day clock can jump backward. So ordering events across machines by timestamp is unreliable — and that quietly corrupts data.[DDIA Ch.8]

📖Jargon decoder
Wall-clock vs. monotonic clock
The wall clock (time of day) is what you'd read off a watch, but it can be reset backward. A monotonic clock only ever moves forward — the right tool for measuring durations (timeouts), not for stamping absolute times.
Skew & drift
Skew = two machines' clocks differ right now; drift = clocks tick at slightly different rates and grow apart.
NTP
The Network Time Protocol syncs clocks over the network — but only to within milliseconds (often worse), and a sync can step a clock backward.[NTP]
🔑Key idea

Don't order events across machines by wall-clock time. Where exact ordering truly matters, use either logical clocks (next topic) or specialized hardware: Google Spanner's TrueTime returns a time interval [earliest, latest] rather than a single instant, and deliberately waits out that uncertainty before committing — keeping it under ~1 ms at the 99th percentile.[Google Spanner]

6.3Ordering without a clock: happens-before & logical clocks

What it is. If we can't trust wall-clocks, how do we order events? Leslie Lamport's answer (1978): track causality — what could have caused what — instead of wall-clock time.[Happened-before]

📖Jargon decoder
Happened-before (a → b)
a definitely came before b if: they're in the same process and a is first; OR a sends a message and b receives it; OR transitively (a→b and b→c ⟹ a→c).[Wikipedia]
Concurrent
If neither a → b nor b → a, the events are concurrent — no one can say which came first (and that's fine).
Lamport timestamp
A simple per-process counter: increment on each event; on receiving a message, set your counter to max(yours, received) + 1. It gives an order consistent with causality — but cannot tell whether two events were concurrent.[Wikipedia]
Vector clock
A richer counter (one per process) that can detect concurrency — at the cost of O(N) size. (Intuition only: know it exists.)[Wikipedia]

6.4The consistency spectrum

What it is. "Consistency" isn't on/off — it's a dial of how up-to-date a read is guaranteed to be, trading strength against latency and availability.[Jepsen]

📖Jargon decoder
Strong consistency / linearizability
The system behaves as if there's a single copy: every read sees the most recent write.[Jepsen]
Causal consistency
Causally-related operations are seen in the same order by everyone (concurrent ones may differ).
Read-your-writes
A client always sees its own prior writes — exactly the fix for the §4.8 replication-lag bug.
Eventual consistency
If writes stop, all replicas eventually converge — but reads may be stale in the meantime, and it promises nothing about when.[AWS]
🌍Real-world scenario

That stale read in §4.8 (you updated your name but saw the old one) was eventual consistency — and "read-your-writes" is the targeted fix. You pick per feature: a bank balance demands strong consistency (never show money that isn't there); a like-count or follower number is fine being eventually consistent for a few seconds. DynamoDB even lets you choose per read request.[AWS]

6.5Network partitions and the CAP theorem

What it is. A network partition is a communication break that splits your nodes into groups that can't reach each other — a network event, not a crashed node.[Wikipedia] The CAP theorem says a distributed system can provide at most two of Consistency, Availability, and Partition-tolerance.

⚠️Watch out — CAP's biggest misconception

CAP is not "permanently pick 2 of 3." Because partitions are unavoidable on real networks, partition-tolerance isn't optional — so the real, narrow trade-off is only this: when a partition is happening, choose Consistency or Availability. When there's no partition, you can have both.[Brewer] Also note CAP's words are narrow: "consistency" here means linearizability, and "availability" has a strict technical meaning. Martin Kleppmann argues the simplistic CP/AP labels should be retired because they mislead more than they help.[Kleppmann]

🌍Real-world scenario

Two data centers lose the link between them and a write arrives. A bank ledger chooses CP — refuse the write rather than risk double-spending. A shopping cart or social feed chooses AP — keep accepting writes and reconcile the divergence later (next topic). Same partition, opposite right answers, driven entirely by what the data is for.

6.6PACELC and resolving conflicts

What it is. PACELC (Daniel Abadi, 2010/2012) completes the picture CAP leaves out: if there's a Partition, trade Availability vs Consistency (that's CAP); Else — in normal operation — trade Latency vs Consistency.[PACELC]

🔑Key idea

PACELC's insight is that the latency-vs-consistency trade-off is present all the time, not just during rare partitions — keeping replicas consistent means waiting for at least one cross-network round trip on every write. Systems get classified by both choices: Dynamo / Cassandra are PA/EL (favor availability and low latency); fully-ACID systems like PostgreSQL are PC/EC (favor consistency, pay the latency and availability cost).[Abadi]

When you pick AP / eventual consistency, conflicts happen — two replicas accept concurrent writes to the same key. How do you reconcile them?

Last-write-wins (LWW)

  • Keep the write with the highest timestamp, discard the rest.
  • Simple — but "dangerously prone to data loss," and it trusts clocks (see §6.2 skew).[ref]

Merge / CRDTs

  • Application merge: reconcile versions so nothing is lost (e.g. union two shopping carts).
  • CRDTs (Conflict-free Replicated Data Types): data structures that merge concurrent updates automatically, deterministically, and without coordination — always converging.[crdt.tech]
🌍Real-world scenario

You add a coat to your cart on your phone and a hat on your laptop during a network blip. With LWW, one device's write wins and you silently lose an item. With an application merge (union the carts) you keep both. Collaborative editors (think real-time docs) and distributed counters lean on CRDTs so everyone's concurrent edits converge to the same result without a central coordinator.

✅Recap — Sections 5 & 6

Caching (§5) serves read-heavy traffic from RAM and the edge so it never touches the database — but it creates extra copies of your data. Distributed reality (§6) is the theory of living with copies: you can't trust clocks, so you order by causality (happens-before); consistency is a dial from strong to eventual; CAP forces a Consistency-vs-Availability choice only during a partition; and PACELC adds the everyday Latency-vs-Consistency trade-off, with LWW / merge / CRDTs to reconcile conflicts. Next, Section 7 asks the natural follow-up: if nodes can't trust clocks and partitions happen, how do they ever agree on anything? That's quorums and consensus.

Section 7 · Distributed theory

Agreement: Quorums & Consensus

Section 6 left us with a hard problem: copies disagree, clocks lie, and partitions happen. So how do a bunch of unreliable machines ever agree on anything — which value is current, who's in charge, what order things happened in? The answer climbs from a simple counting trick (quorums) to one of computer science's deepest results (consensus).

7.1Quorums and the R + W > N rule

What it is. A quorum is requiring a minimum number of replicas to agree before an operation counts. With N replicas, you write to W of them and read from R of them.[Cassandra]

🔑Key idea — a tunable dial

R and W are knobs on the consistency/speed trade-off from §6.4. Small (W=1) → fast, available writes but weak consistency; large (W=N) → strong but slow and fragile (one slow/down node blocks you). With N=3, W=2, R=2 you can lose one node and still read and write. If R + W ≤ N, you trade the freshness guarantee for more speed and availability.[Cassandra]

⚠️Watch out

R + W > N guarantees your read overlaps a fresh copy — it does not by itself give full linearizability. Concurrent writes still need conflict resolution (§6.6), and the "sloppy quorum" below deliberately breaks the overlap to stay available. Read it as "you'll see at least one up-to-date copy," not "perfectly consistent."

7.2Staying available: sloppy quorums and convergence

The problem. A strict quorum fails if too many of the specific nodes that own a key are unreachable. To stay available during failures, Dynamo-style systems bend the rules.

📖Jargon decoder
Sloppy quorum + hinted handoff
During a partition, accept the write on the first N healthy nodes instead of the exact owners; a stand-in stores the data with a hint and hands it off to the real owner once it recovers. ("Sloppy quorum" is the Dynamo paper's term; Cassandra calls the mechanism hinted handoff.)[Dynamo]
Read repair
During a read, if the contacted replicas disagree, fix the stale ones right then, in the read path.[Cassandra]
Anti-entropy
A background process where replicas build Merkle trees (hash trees) of their data and compare them to find and re-sync mismatched ranges.
🌍Real-world scenario

A replica that owns your shopping-cart key goes down. Instead of failing your write, the system stores it on a healthy stand-in node with a hint. When the owner comes back, the hint is handed off; meanwhile read repair and anti-entropy quietly heal any divergence. You never saw an error — availability was preserved at the cost of brief inconsistency (an AP choice, §6.5).

7.3The consensus problem (and why it's hard)

What it is. Consensus is getting several nodes to agree on a single value — or, in practice, on an ordered sequence of values. It's the foundation under leader election, distributed locks, and replicated logs. Agreement requires a majority — more than half — so to tolerate f failures you need 2f + 1 nodes (3 nodes survive 1 failure; 5 survive 2).[DDIA Ch.9]

⚠️Watch out — the FLP impossibility result

The famous FLP result (Fischer, Lynch, Paterson, 1985) proves that in a fully asynchronous network where even one node may crash, no deterministic algorithm can guarantee that consensus always terminates.[JACM 1985] Crucially, it never breaks safety (the system won't decide two different values) — only liveness (it might not decide at all) can't be guaranteed. Real systems sidestep it with timeouts (assume "too slow = failed") and randomization, so they terminate in practice. The lesson: perfect agreement on an unreliable network has a fundamental limit.

7.4Raft: consensus you can actually understand

What it is. Raft is a consensus algorithm explicitly designed to be more understandable than the older Paxos, while being equally fault-tolerant.[Raft] It elects a single leader that takes all writes and replicates an append-only log to the followers.

🌍Real-world scenario

The leader of a 3-node cluster crashes. The remaining two start a new term and elect a new leader (2 votes = majority). If the old leader comes back, it can't commit anything — it can't get a majority — so it steps down. No split-brain, no two "current" values. This is exactly how etcd and Consul keep a cluster's state consistent (next topic).

7.5Coordination services & service discovery

What it is. You rarely implement Raft yourself — you use a coordination service: a small, strongly-consistent, replicated store built for exactly these jobs (leader election, distributed locks, configuration, and service discovery). ZooKeeper (uses the Zab protocol), etcd (uses Raft, and stores Kubernetes' entire state), and Consul are the common ones; each stays available as long as a majority of its servers are up.[etcd] [ZooKeeper]

🔑Key idea

Consensus isn't an abstract puzzle — it's a service you depend on. This same strongly-consistent substrate is what lets the microservices in Section 10 find each other dynamically as instances come and go. "Where is service X right now?" is answered by a consensus-backed registry.

✅Section 7 recap

Quorums (R + W > N) are a cheap counting trick that guarantees a read overlaps a fresh write; loosen them (sloppy quorums, hinted handoff, read repair, anti-entropy) to stay available. Consensus is the deeper problem of agreeing on a value or an ordered log, always needing a majority — and FLP says perfect agreement can't be guaranteed on an async network, so we use timeouts. Raft makes consensus understandable (one leader, a replicated log, majority commit, no split-brain), and coordination services (etcd/ZooKeeper/Consul) package it for leader election and service discovery. Next: even with agreement, individual requests can still be lost, duplicated, or half-applied — Section 8 is making the application correct.

Section 8 · Distributed theory

Application Correctness: Idempotency, Delivery & Transactions

Consensus keeps the cluster agreeing — but your application still issues individual calls that can be lost, duplicated, or half-finished. This section is the everyday toolkit for correctness under failure: never charge a card twice, never lose an event, and never leave a multi-service operation half-done. These are the patterns you'll reach for constantly.

8.1The three outcomes of a remote call

What it is. Every call across a network has not two but three possible outcomes: success, failure, and the dreaded unknown — you got no response and cannot tell whether it worked (§6.1, made personal).[Stripe]

8.2Idempotency: making retries safe

What it is. An operation is idempotent if doing it many times has the same effect as doing it once — which is exactly what makes a retry safe. Per RFC 9110, GET, HEAD, PUT, DELETE, and OPTIONS are idempotent; POST is not.[RFC 9110] So how do you make a POST (like "charge this card") safe to retry?

📖Jargon decoder
Idempotency key
A unique client-generated ID (Stripe uses an Idempotency-Key header, recommending a random UUID) sent with a request so the server can deduplicate retries.[Stripe]
Natural vs. synthetic idempotency
Some ops are naturally idempotent ("set balance to 100", a PUT); others (create/charge) must be made idempotent synthetically with a dedup key.
🌍Real-world scenario

A user taps "Pay" and the network drops the response. The app retries with the same idempotency key. Because the server already processed that key, it returns the original receipt and does not charge again. (Pair this with capped exponential backoff + jitter and a timeout on every call.[AWS])

8.3Delivery semantics & the "exactly-once" myth

What it is. When systems pass messages, there are three possible guarantees:

SemanticGuaranteeIn practice
At-most-onceMay be lost, never duplicatedFire-and-forget; fine for disposable data (some metrics)
At-least-onceNever lost, may be duplicatedThe common default (Kafka, Pub/Sub); requires idempotent consumers[Kafka]
Exactly-onceEach effect happens onceThe ideal — but read the warning below
⚠️Watch out — exactly-once delivery vs. processing

Exactly-once delivery over an unreliable network is generally considered impossible (it's the Two Generals problem — retries are unavoidable, so duplicates can always happen).[Pub/Sub] But exactly-once processing is achievable: combine at-least-once delivery with idempotency/dedup (§8.2) so duplicates are harmless. When Kafka advertises "exactly-once," it means exactly-once processing within Kafka (idempotent producers + transactions between topics) — not a magic guarantee over the open internet.[Kafka]

8.4Distributed transactions: 2PC and sagas

The problem. The single-machine ACID transaction from §4.2 does not stretch across multiple services or databases — and in microservices each service owns its own database.[Microsoft] Two patterns address this.

Choreography

  • No central brain: each service does its local step and emits an event that triggers the next.
  • Simple, decoupled — but the overall flow is implicit and harder to trace.[Microsoft]

Orchestration

  • A central orchestrator tells each service what to do, tracks progress, and drives compensations on failure.
  • Easier to see and manage — but the orchestrator is a component to build and run.
🌍Real-world scenario

An order spans three services: Order, Payment, Inventory. As a saga: create the order → take payment → reserve stock. If reserving stock fails (out of stock), the saga runs compensations: refund the payment, cancel the order. No service ever held a lock on another, and the system ends in a consistent state — just not instantaneously.

8.5The outbox pattern: the dual-write problem

The problem. A service often must do two things together: update its database and publish an event ("OrderPlaced") to a message broker. But those are two separate systems — you can't write to both atomically. A crash in between leaves them inconsistent: the order saved but the event lost (or vice versa). This is the dual-write problem.[microservices.io]

🔑Key idea — it all connects

The outbox gives at-least-once delivery (the relay might publish a duplicate after a crash), so consumers must be idempotent (§8.2). Notice the whole section forms one chain: the network's unknown forces retries → retries need idempotency → reliable messaging is at-least-once → which leans on idempotency again → and sagas + the outbox build multi-service correctness on top. Idempotency is the keystone.

✅Recap — Sections 7 & 8

Agreement (§7): quorums (R + W > N) cheaply guarantee a fresh read; consensus needs a majority and is bounded by FLP; Raft makes it understandable (leader + replicated log, no split-brain); coordination services package it for leader election and service discovery. Correctness (§8): every remote call has a third outcome — the unknown — forcing retries, which demand idempotency; messaging is realistically at-least-once (exactly-once processing, not delivery); and multi-service operations use sagas with compensations instead of fragile 2PC, with the outbox pattern making "update DB + publish event" reliable. Next, Section 9 builds the communication layer these patterns ride on: messaging, APIs, and protocols.

Section 9 · Building blocks

Communication: Async Messaging, APIs & Protocols

Sections 6–8 were about surviving the hard truths of distributed systems. This section is the everyday plumbing that connects your services: how to hand off slow work without blocking users, how to broadcast events, and which flavor of API or real-time channel to reach for. It's the layer the correctness patterns from §8 ride on.

9.1Synchronous vs. asynchronous, and message queues

What it is. In synchronous communication the caller waits (blocks) for a response; in asynchronous communication the caller hands off the work and continues, getting the result later.[Microsoft] Blocking a user on slow work — sending email, encoding video, generating a report — is bad UX and couples your availability to that slow service.

📖Jargon decoder
Message queue
A buffer between a producer and a consumer (e.g. SQS, RabbitMQ). It decouples them — they don't need to be available at the same time — and smooths load spikes ("queue-based load leveling").[Microsoft]
Background worker
A separate process that pops jobs off the queue and executes them, so the web request can return instantly.[RabbitMQ]

✓ Use a queue when

  • Work is slow, spiky, or can happen later (email, encoding, exports).
  • You want producers and consumers to fail independently.

✗ Don't when

  • The caller genuinely needs a low-latency, synchronous answer.
  • Volume is low, stable, and simple — a queue is just overhead.[Microsoft]

9.2Publish–subscribe and event-driven thinking

What it is. A queue delivers each message to one consumer. Publish–subscribe flips that: a publisher sends to a topic, and every interested subscriber gets its own copy — one-to-many fan-out.[AWS SNS] This enables event-driven design: services emit events ("OrderPlaced") and others react, without the publisher knowing who's listening.

⚠️Watch out

Fan-out's loose coupling has a cost: the end-to-end flow is harder to trace (attach a correlation ID to every message so you can follow it — foreshadowing §11), and pub/sub is asynchronous and eventually consistent with no guaranteed cross-subscriber order.[Microsoft] Managed options: Amazon SNS, Google Pub/Sub, Azure Service Bus/Event Grid.

9.3Event streaming with Kafka

What it is. Apache Kafka is a distributed event streaming platform — really an append-only, durable log of events.[Kafka] A topic is split into partitions; within a partition, records are strictly ordered and each has an offset.

🔑Key idea — a log you can replay

Two things make Kafka different from a plain queue: ordering is guaranteed only per partition (not across the whole topic), and Kafka retains events for a configured time so consumers can re-read / replay them — events aren't deleted on consume.[Kafka] Replay lets you rebuild a downstream store or onboard a new consumer from history — the backbone of event-driven data pipelines.

9.4Backpressure, consumer lag & dead-letter queues

What it is. Async systems fail in specific ways you must plan for.

📖Jargon decoder
Backpressure
When consumers can't keep up with producers, the system must slow producers, buffer, or shed load — otherwise it falls over. (Uncoordinated aggressive retries cause a "retry storm" that turns a small problem into a cascade.)[Microsoft]
Consumer lag
How far behind the latest message a consumer group is — the single most important health metric for a stream.[Confluent]
Dead-letter queue (DLQ)
A side queue where messages that fail repeatedly (a "poison message") are parked after a max retry count, so one bad message doesn't block the queue — and you can inspect it later.[AWS]

9.5REST API design

What it is. REST organizes an API around resources named by URLs (nouns, not verbs), acted on with the HTTP verbs and status codes from §1.7.[Microsoft] The habits that matter in practice:

  • Resources & verbs: GET /orders/123 (read), POST /orders (create), PUT (replace), PATCH (partial update), DELETE. Collections are plural nouns (/orders).
  • Pagination: never return a giant collection at once. limit/offset is simple, but cursor-based pagination scales better — offset must scan and discard all skipped rows, which gets slow on big, changing datasets.[Gusto]
  • Versioning: evolve without breaking clients — commonly a path prefix (/v1/…). Adding fields is safe if clients ignore unknown ones; removing/renaming is breaking.
  • The contract: the agreed interface, described with OpenAPI (formerly Swagger), so clients and servers (and docs and tests) stay in sync.
🌍Real-world scenario

Your mobile app calls GET /v1/products?limit=20&cursor=…. Months later you add a discount field — existing clients ignore it, so nothing breaks. When you must make a breaking change, you ship /v2/ and keep /v1/ alive during migration. The OpenAPI spec is the single source of truth both teams build against.

9.6RPC, gRPC & GraphQL — beyond REST

What they are. REST isn't the only way for programs to talk. Two big alternatives:

📖Jargon decoder — the problem GraphQL solves

Over-fetching = an endpoint returns more than you need (wasted bytes). Under-fetching = one endpoint doesn't return enough, forcing several round-trips. GraphQL lets the client ask for precisely the right shape in one request — great for mobile, at the cost of more server-side complexity and harder caching.[GraphQL]

9.7Real-time delivery: pushing to the client

The problem. Plain HTTP is client-initiated — the server can't spontaneously push. Four ways around it, from crude to purpose-built:

🔑Key idea — pick the simplest that fits

If data only flows server → client (a notification feed, a live score), SSE is simpler than WebSockets and auto-reconnects. If both sides push in real time (chat, a shared document, a game), you need a WebSocket. Don't reach for a WebSocket when SSE — or even long polling — would do.

9.8API gateways: the single front door

What it is. An API gateway is one entry point that sits in front of many backend services and handles the cross-cutting concerns so each service doesn't have to: routing to the right service, authentication, TLS termination, rate limiting, request aggregation, and observability.[Microsoft]

⚠️Watch out

Because everything flows through it, an API gateway can become a bottleneck or single point of failure — run several instances behind a load balancer so it's highly available (the §3 lesson again).[Microsoft] A common variant is Backend-for-Frontend (BFF): a tailored gateway per client type (one for the mobile app, one for the web app).

9.9Rate limiting at the edge

Why it exists. Rate limiting caps how fast a client can call you — to stop abuse/DoS, ensure fairness between clients, and protect finite capacity and cost.[Stripe] (This is where the Redis distributed-counter idea from §5 finally pays off.)

✅Recap — Section 9

Hand off slow work with queues (decouple + load-level) and background workers; broadcast with pub/sub fan-out; stream and replay with Kafka (per-partition order, consumer groups); survive failure with backpressure, lag monitoring, retries, and dead-letter queues. Expose it all through the right API — REST for public CRUD, gRPC for fast internal calls, GraphQL for flexible clients — push in real time with WebSockets/SSE, and front everything with an API gateway that handles auth, routing, and rate limiting (429 + a shared Redis counter). Section 10 · Higher altitude

Architecture, Resilience & the Deployment Substrate

Now we zoom out from individual components to how the whole system is arranged and run: monolith or microservices, the patterns that keep it standing when parts fail, and — at last — Docker and Kubernetes, the words you started this guide not understanding. This section arranges everything you've learned into something you can actually ship and operate.

10.1Monolith first — and what pushes you to split

What it is. A monolith is a single deployable unit containing all the app's functionality, whose parts talk in-process (not over the network) and typically share one database.[AWS]

🔑Key idea — start with a monolith

Martin Fowler's advice is blunt: don't start a new project with microservices, even if you're sure it'll get big enough to justify them. Almost all successful microservice systems started as a monolith that got too big and was broken up; ones built as microservices from scratch often ended in trouble — because getting stable service boundaries right up front is extremely hard.[Fowler] Build the monolith first, find the natural seams, then split. (This is a strong default, not universal law — some argue for splitting early.)

What pushes teams to split: independent scaling of a hot component, team autonomy with clear ownership (Conway's law — services aligned to teams), and faster independent deploys so one team doesn't wait on another.[microservices.io]

10.2Microservices — and the cost they add

What it is. Microservices structure an app as a suite of small, independently deployable services, each running in its own process, communicating over the network, and — crucially — each owning its own database (decentralized data, accessible only via that service's API).[Fowler] [microservices.io] You decompose around bounded contexts / business capabilities, not technical layers.

⚠️Watch out — microservices aren't free

Fowler is explicit that "distributed software has a major disadvantage: the fact that it's distributed." Remote calls are slow and can fail, so with many services there are far more failure points — you must design for partial failure (§6). And because each service owns its data, a business transaction spanning services can't use one ACID transaction — you fall back to eventual consistency and sagas (§8), plus heavy operational overhead. Pay this "microservice premium" only when the benefits are worth it.[Fowler]

10.3Resilience I: timeouts, deadlines & retries

What it is. Once calls cross the network, you must plan for them being slow or failing. Three habits form the base layer.

📖Jargon decoder
Timeout
Every remote call needs one — without it, a hung downstream service makes in-flight requests pile up until the caller runs out of resources. Pick it from a high latency percentile of the callee.[AWS]
Deadline propagation
Set one absolute deadline at the top of the stack and pass a shrinking remaining budget down the call chain, so a downstream call never runs longer than the caller will wait — and each stage stops early if the deadline already passed.[Google SRE]
Backoff + jitter
Retries wait progressively longer (capped exponential backoff) with added randomness (jitter) so many clients don't retry at the same instant.[Google SRE]
⚠️Watch out — the retry storm

Retries stacked at every layer multiply: 3 retries (4 attempts) at each of 3 layers can become 4³ = 64 attempts hammering the database — a "retry storm" that turns a small hiccup into a cascade.[Google SRE] So: retry at one layer, cap attempts, only retry idempotent (§8) operations on genuinely transient faults, and keep total retry time within your deadline.

10.4Resilience II: circuit breakers, bulkheads & degradation

What it is. The patterns that contain a failure before it spreads into a cascading failure (one overloaded part making its callers pile up and fail, propagating outward).[Google SRE]

📖Two more moves under overload
Load shedding
Deliberately reject a proportion of traffic as you approach overload (e.g. return 503) to protect the system rather than collapsing.[Google SRE]
Graceful degradation
Serve reduced or partial functionality instead of failing entirely — e.g. show cached results, or hide a non-essential recommendation panel, when a dependency is down.
🌍Real-world scenario

Your checkout calls a flaky fraud-scoring service. A circuit breaker trips after repeated failures, so checkout stops hanging on it and degrades gracefully — accepting the order for manual review instead of blocking. A bulkhead ensures the fraud service's stuck calls can't consume every thread and take the whole checkout down with it. (Popular libraries: resilience4j, Polly.)

10.5Serverless & Functions-as-a-Service

What it is. Serverless means you don't provision or manage servers — the platform runs your code on demand, auto-scales (including to zero when idle), and you pay per use (per request + execution time), not for idle capacity.[AWS Lambda] [Google] FaaS (e.g. AWS Lambda) runs individual event-triggered functions.

✓ Great for

  • Spiky, low, or event-driven workloads (an upload triggers a resize; a queue message triggers a job).
  • Avoiding all server ops and paying nothing when idle.

✗ Poor fit / watch out

  • Cold starts — added latency when a function spins up from idle.
  • Max execution time (AWS Lambda caps at 15 minutes), enforced statelessness, and vendor lock-in.
  • Not ideal for steady high-throughput or long-running work.[AWS]

10.6Containers & Docker, in depth

What it is. Back in §1.5 a container was "a lightweight box sharing the host kernel." Here's the Docker workflow that makes it real. A Dockerfile (a recipe) builds an image — an immutable template, stored in read-only layers, bundling your code + runtime + dependencies. A container is a running instance of an image. Images are shared via a registry (Docker Hub, ECR).[Docker]

The image format and runtime are standardized by the OCI (Open Container Initiative), so containers aren't locked to one vendor.[OCI]

10.7Kubernetes: orchestrating containers

What it is. Once you have hundreds of containers across many machines, you need an orchestrator. Kubernetes (K8s) automates deploying, scaling, and operating containerized apps across a cluster.[Kubernetes] It's really just automation for the instances, health checks, and horizontal scaling you already understand from §3.

📖Jargon decoder
Pod
The smallest deployable unit — one or more containers sharing network and storage.[K8s]
Deployment
Declares the desired state (which image, how many replica pods); a controller continuously reconciles actual → desired.[K8s]
Service
A stable virtual IP / DNS name that load-balances across a changing set of pods (§3's load balancer, built in).[K8s]
HorizontalPodAutoscaler
Scales the replica count up/down based on metrics (CPU, etc.) — §3's autoscaling.

10.8Deployment strategies

What it is. How you ship a new version without downtime or big-bang risk.

StrategyHow it worksRollback
RollingReplace instances gradually, a few at a time (Kubernetes' default)Roll the update backward
Blue-greenTwo identical environments (blue = live, green = new); switch all traffic at once[Fowler]Instant — switch back to blue
CanarySend the new version to a small % of traffic first, watch metrics, then ramp up[Fowler]Route the canary back to stable
📖Two enablers
Feature flags (toggles)
Decouple deploy from release — ship code "dark" and turn a feature on at runtime, no redeploy.[Fowler]
Expand/contract migrations
Make a breaking schema change safe in three steps: expand (add the new schema alongside the old), deploy code that works with both + backfill, then contract (remove the old) — so a migration never breaks running code.[Fowler]

10.9Infrastructure as code, immutable infra & environments

What it is. The last piece: managing the infrastructure itself like software.

📖Jargon decoder
Infrastructure as Code (IaC)
Define and provision infrastructure through version-controlled, declarative code (Terraform, CloudFormation) instead of manual console clicks — making environments reproducible, reviewable, and automatable.[Terraform]
Immutable infrastructure
Never modify a running server in place; to change it, build a new image/instance and replace the old one. This kills configuration drift and "snowflake servers" (unique, hand-tweaked, unreproducible boxes).[Fowler]
Environments
Separate dev / staging / prod that mirror each other, with configuration kept in the environment, not baked into code (12-factor).[12-factor]
🌍Real-world scenario

Your whole stack — networks, load balancers, the Kubernetes cluster — is described in Terraform in git. A change is a pull request, reviewed like code. To deploy, you build a fresh container image and let Kubernetes replace the old pods (immutable) — nobody ever SSHes in to patch a live server. Spin up an identical staging environment from the same code to test first. That's the modern operational baseline.

✅Recap — Section 10

Start with a monolith; split into microservices (own database each) only when scaling/team/deploy pressure justifies the distributed cost. Keep it standing with resilience patterns — timeouts, deadline propagation, bounded retries with backoff+jitter, circuit breakers, bulkheads, load shedding, and graceful degradation — to stop cascading failures. Run it on the right substrate: serverless for spiky/event work, containers (Docker: build → ship → run) packaged once and run anywhere, and Kubernetes to orchestrate them with declarative, self-healing desired-state. Ship safely with rolling/blue-green/canary deploys, feature flags, and expand/contract migrations, all defined as immutable infrastructure as code. Section 11 · Operating it

Operating It: Observability, Autoscaling, Reliability & Security

A system that only exists in a diagram is easy. This section is about running the real thing: seeing inside it, scaling it automatically, proving how reliable it is, surviving disasters, controlling cost, and keeping it secure — plus the human practices that keep it all sane. These cross-cutting concerns protect and operate everything you've built.

11.1Observability: the three pillars

What it is. Observability is the ability to understand a system's internal state from its external outputs — to ask new questions you didn't anticipate — whereas plain monitoring watches predefined signals.[OpenTelemetry] It rests on three complementary pillars:

📖Jargon decoder
Logs
Timestamped records of discrete events. Use structured logging (JSON key-value, so every field is queryable) and log aggregation (ship them to a central searchable store — ELK, Loki).[OTel]
Metrics
Numeric time-series aggregated over time (request rate, error rate, latency, CPU). Compact and cheap → the foundation of dashboards and alerts.
Traces
Distributed tracing follows one request across many services as a tree of spans — stitched together by a correlation / trace ID propagated through every hop (§9).

11.2What to measure — and how to alert

What it is. You can't watch everything, so measure the signals that predict user pain. Three well-known recipes, all rooted in Google's Four Golden Signals:

FrameworkMeasureBest for
Four Golden SignalsLatency · Traffic · Errors · SaturationAny user-facing system[Google SRE]
REDRate · Errors · DurationRequest-driven services[Grafana]
USEUtilization · Saturation · ErrorsResources (CPU, disk, pools)[Gregg]
⚠️Watch out — alert fatigue

Page on symptoms (users are impacted / the SLO is burning), not on every raw cause. Too many noisy, non-actionable alerts cause alert fatigue — on-call engineers go numb and miss the real incident.[Google SRE] Every page should be urgent, actionable, and user-impacting; everything else belongs on a dashboard.

11.3SLI, SLO, SLA & error budgets

What it is. This is how you make the non-functional requirements from §1.8 measurable and enforceable.

📖Jargon decoder
SLI — Service Level Indicator
A measured quantity of service level, e.g. the proportion of requests served in under 300 ms, or the success rate.[Google SRE]
SLO — Service Level Objective
A target for an SLI over a window, e.g. 99.9% success over 28 days — your internal goal.
SLA — Service Level Agreement
A contract with customers that has consequences (refunds) if missed. Set a tighter internal SLO than what you advertise, so the external SLA ends up looser — you get warned before you break a promise.
Error budget
1 − SLO. A 99.9% SLO permits 0.1% unreliability. It's a budget you're allowed to spend.[Google SRE]

11.4Autoscaling — closing the loop

What it is. Back in §3 we scaled out manually. Autoscaling does it automatically, driven by the metrics from §11.1 and the health checks from §3. Flavors: horizontal (add/remove instances — the common one), vertical (resize an instance), scheduled (known time patterns), dynamic / target-tracking (keep a metric like CPU at a target), and predictive (forecast demand).[AWS] [K8s HPA]

11.5Availability math: the nines, and why dependencies hurt

What it is. Availability is quoted in "nines," each nine cutting allowed downtime ~10×:

AvailabilityName≈ Downtime / year
99%two nines~3.65 days
99.9%three nines~8.8 hours
99.99%four nines~53 minutes
99.999%five nines~5 minutes

Figures are approximate (they vary slightly with a 365- vs 365.25-day year).

🔑Key idea

A request that touches ten services, each "four nines," is only about three nines overall — dependencies drag availability down. So you buy it back by removing single points of failure with redundancy (the load balancer and database SPOFs from §3–4). High availability = no SPOF + enough nines where it counts.

11.6Failure domains, disaster recovery & RPO/RTO

What it is. A failure domain (or blast radius) is the scope one failure can take out — a pod, an AZ, a whole region (§2). Disaster recovery plans for the big ones, against two targets:

📖Jargon decoder — the two DR targets
RPO — Recovery Point Objective
Maximum acceptable data loss, measured in time (how far back you'd lose data). Smaller RPO → replicate more often.[AWS DR]
RTO — Recovery Time Objective
Maximum acceptable downtime to restore service. Smaller RTO → warmer, pricier standby.
DR strategyWhat it isCost / speed

Pilot LightData replicated; core infra provisioned but "off"↓
Warm StandbyScaled-down but always-running copy in another region↓

The four AWS DR strategies, from lowest cost/slowest recovery to highest cost/fastest.[AWS]

11.7Capacity planning, load testing & cost

What it is. Capacity planning is provisioning enough for forecast demand plus headroom for spikes and failover — never run at 100%.[Google SRE] You verify capacity by testing:

TestPurpose
Load testConfirm the system handles expected peak traffic
Stress / breakpointPush past limits to find the actual breaking point
Spike testSurvive a sudden, short traffic surge (a launch, a sale)
Soak testRun for hours/days to reveal leaks and slow degradation

Common tools: k6, JMeter, Gatling, Locust.[Grafana]

🔑Key idea — cost is a first-class constraint

There's a real tension: over-provisioning wastes money, under-provisioning causes outages. Modern practice adopts a consumption model — pay for what you use, autoscale to demand, right-size instances, and mix on-demand / reserved / spot pricing.[AWS] A "correct" design that's 5× too expensive is the wrong design.

11.8Security: identity, encryption & secrets

What it is. Security is a design constraint woven through everything, not a bolt-on. Start with the pair everyone confuses:

📖Jargon decoder
JWT & OAuth 2.0
A JWT is a compact, digitally-signed token carrying claims (so the server can trust it without a lookup). OAuth 2.0 (RFC 6749) is a delegated authorization framework that issues access tokens — the "Sign in with Google" flow. (Both build on the cookies/tokens from §3.)[OAuth 2.0]
Encryption in transit vs. at rest
In transit = TLS while data moves (§2); at rest = encrypting stored data on disk/in the DB. You need both.[AWS]
Secrets & key management
Never hardcode credentials. Store them in a secrets manager / Vault, protect encryption keys in a KMS, and rotate them regularly to shrink the blast radius of a leak.[Vault]

What it is. A few threats every engineer should recognize, and the constraints regulations put on architecture.

🌍Real-world scenario — compliance shapes architecture

Regulations like GDPR restrict where and how personal data may be stored and processed (data residency / sovereignty). If EU users' data must stay in the EU, that decides which region (§2) it lives in — a legal requirement reaching all the way down into your infrastructure design.[GDPR]

11.10The human side: on-call, toil & postmortems

What it is. Systems are run by people, and how you run them decides whether they stay healthy.

📖Jargon decoder
On-call
A rotation responsible for responding to production alerts within a target time (roughly ~5 min for critical, ~30 min for less urgent services).[Google SRE]
Toil
Manual, repetitive, automatable work with no enduring value that scales with the service. Google's SRE model caps it (~50% of time) and automates it away — otherwise operational load buries the team.[Google SRE]
Blameless postmortem
A written incident retrospective focused on systemic causes and prevention, not blaming individuals — so engineers report honestly and the organization actually learns.[Google SRE]
🌍Real-world scenario

An outage wakes the on-call engineer; they mitigate, then write a blameless postmortem: what happened, why the system allowed it, and action items — never "whose fault." One action item is usually to automate away the toil that caused or worsened it. Do this consistently and the same incident doesn't recur.

✅Recap — Section 11

See inside the system with the three pillars (logs, metrics, traces tied by a trace ID); measure the golden signals and alert on symptoms, not noise. Make reliability enforceable with SLIs/SLOs/SLAs and error budgets that govern release speed. Autoscale off those metrics (closing §3's loop), understand that dependencies multiply availability down while redundancy multiplies it up, and plan for disasters with RPO/RTO and the right DR strategy. Right-size for cost, verify with load testing, and secure it end-to-end — authN/authZ, least privilege, encryption in transit and at rest, managed secrets, and defenses against DDoS/injection under compliance constraints — all run by humans with sane on-call, minimal toil, and blameless postmortems. 12.1A repeatable design method — and a symptom→cause map

12.2Back-of-the-envelope sizing, in context

What it is. The estimation primer from §1.10, now used against a real target. Four formulas cover most of it:

QuantityFormula
Average QPSrequests per day ÷ 86,400
Peak QPS≈ 2 × average QPS (rule of thumb)

🌍Worked example — a social feed

At ~300 bytes of text per post, that's ~90 GB/day, or ~164 TB over 5 years (before media, which goes to object storage anyway). In seconds you've learned this is write-heavy at the tweet layer but read-heavy at the feed layer — which decides the whole design. (Numbers are illustrative assumptions, not facts — that's the point of BOTE.)

  • 302 (temporary) → every click reaches you, enabling analytics.[MDN]

12.4Capstone: a news feed (Twitter / Instagram)

12.5Capstone: a chat system (WhatsApp / Messenger)

The core need: instant, bidirectional delivery — so the transport is a persistent WebSocket (§9), not polling.[ByteByteGo]

Reuses: WebSockets (§9), message queues (§9), a session registry to know which gateway holds each user's connection — a coordination-service job (§7), and per-conversation ordering via sequence IDs (§8).

12.6Capstone: video streaming (YouTube / Netflix)

The core idea: you don't stream "a video file" — you stream small chunks at a quality chosen live for each viewer's bandwidth.

Reuses: object/blob storage (§4), async background workers for transcoding (§9), and CDN edge delivery (§5) — the media section's whole reason to exist.

12.7Capstone: ride-sharing / "find nearby drivers" (Uber)

The core challenge: find drivers near a rider without computing the distance to every driver on Earth. The answer is a geospatial index.

Reuses: WebSockets for the flood of location pings (§9), an in-memory hot store for latest positions (§5), and sharding to absorb the very high write rate (§4). The geospatial index is the one genuinely new technique — everything around it you already know.

12.8Finding bottlenecks & the operational review

Finding bottlenecks. A design is never "done" — you look for the constraint. Measure it (§11 observability), find the most saturated resource, and it's usually one of a few suspects: the single primary database (§4), a hot key (§5), or a slow synchronous call (§9). The fixes are the tools you've collected: cache it, replicate reads, shard writes, make it async, or add capacity.

🔑The operational review — a design isn't done until you can run it

Before you call any design finished, run it through Section 11: what are its SLOs, how do you deploy it safely (§10), and can you observe it when it breaks? A beautiful architecture you can't operate, secure, or afford is not a good design. The best engineers close this loop every time.

12.9When NOT to distribute

The most important lesson last. This entire guide taught you how to distribute a system — so here's when not to. A single powerful machine, or a well-built monolith with a managed database, handles far more load than beginners assume, and it does so with none of the hard problems you met in Sections 6–8: no partial failure, no distributed consistency, no consensus, no operational sprawl.[Fowler]

⚠️Complexity is a cost you pay forever

Every technique here — replicas, shards, microservices, queues, multi-region — buys scale or resilience at the price of complexity. Adopt each one only when a real, measured requirement demands it, never speculatively (YAGNI). As Fowler puts it, microservices carry a "premium… only useful with more complex systems." You don't have a scaling problem until you have a scaling problem — and the best design is the simplest one that meets the requirements.[Kleppmann]

🎓You made it — from one request to a distributed system

You started not knowing what a "server" was. You now understand the whole arc: a single request hitting one program; scaling it out behind a load balancer with stateless servers; a data layer that's durable, replicated, and sharded; caching and CDNs for speed; the hard truths of distributed systems (clocks, consistency, CAP); consensus and idempotency for correctness; messaging and APIs to connect services; Docker & Kubernetes to run them; observability, SLOs, and security to operate them — and the judgment to know when not to reach for any of it. The words you began not understanding — Docker, Kubernetes, load balancer, instance, server — are now tools you can reason about and choose deliberately. That's system design. Go build something.
